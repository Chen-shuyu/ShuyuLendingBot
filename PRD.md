📋 一、 專案背景與初始評估
1. 經典開源專案（MikaLendingBot）的經驗借鑑
核心日常運作： 機器人採取無限循環機制，包含自動劃轉閒置資金、清理未成交舊單、依市場狀況重新掛單，以及定時休眠。

放棄直接修改的原因： 該專案底層仍綁定已淘汰的 Python 2 語法，且核心通訊層使用舊版 Bitfinex V1 API，未來隨時有被交易所關閉的維護隱患。此外，其原先使用的 LINE Notify 通知管道已於 2025 年 3 月底被官方全面終止。

最終決策： 取其商業邏輯精華，去其舊程式碼糟粕。開闢全新 Python 3 專案，改採現代化的 ccxt 套件封裝對接 Bitfinex V2 API。

2. 本地開發與硬體重生策略
Windows 與 Linux 區網無縫開發： 將家中的舊 Windows 電腦重灌為輕量、無圖形介面的 Linux 伺服器，並插上實體網路線確保通訊穩定。現役 Windows 電腦則安裝 VS Code，利用 「Remote - SSH」 擴充套件，透過區域網路遠端連線至 Linux 伺服器進行開發。

常駐運維： 未來正式上線後，捨棄手動執行，將主流程封裝為 Linux 系統的 systemd 常駐服務，設定崩潰自動重啟機制，確保 24 小時不間斷運作。

🎯 二、 核心策略：FRR+ 雙發彈夾鎖利策略
針對您目前放在 Bitfinex 的 344.12 美金 初始資金，結合交易所的硬性規定，量身打造最適合的翻身模式：

1. 資金限制與拆單邏輯
最低放貸門檻： Bitfinex 規定單筆放貸金額不得低於 150 美元。

雙發彈夾配置： 由於資金總量約為 344 美元，在符合門檻的前提下，策略採「平分兩筆（每筆約 172 美元）」掛出。這能將資金利用率推至最高，且能錯開還款時間，避免資金同時被鎖在低利區間，保留隨時回收資金搶高利的彈性。

2. 利率定價與天期控制
FRR+ 動態底線： 實時抓取 Bitfinex 官方計算的市場平均利率——閃電返還率（FRR）。掛單利率以 FRR 額外加上微幅年化溢價 作為基準，確保收益始終高於市場均價，且享有系統隨市場動態調整的優勢。

高利動態鎖天期（xDays）： 平時市場利率平穩時，一律掛最短的 2 天天期 以維持資金靈活性。一旦機器人偵測到市場年化利率暴飆突破 30%（日利率 0.00082） 的暴利閾值，立即啟動長天期策略，直接掛滿 30 天，強行鎖住高額利息回報。

安全防禦機制： 申請的 API Key 權限嚴格禁止勾選「提現（Withdraw）」，僅開放融資掛單與讀取權限，確保資金在絕對安全的閉環中運行。

🏗️ 三、 未來程式架構規劃 (Future Architecture)
為了實現模組化、可擴充性，並方便您未來自由抽換或調整策略，專案將採取解耦（Decoupling）的結構設計：

1. 專案目錄結構 (Project Directory)
未來完整的專案目錄將依功能劃分如下：

bfx-lending-bot/（專案根目錄）

config/（配置資料夾）

config.yaml —— 統一存放 API Key、LINE 憑證、高利鎖天期閾值等參數，禁止硬編碼在程式中。

modules/（核心功能模組）

exchange_client.py —— 負責與 ccxt 互動，封裝查詢融資可用餘額、取消未成交舊單、發送新放貸單的底層行為。

lending_strategy.py —— 策略大腦，專門負責抓取 FRR、計算小資族雙發彈夾金額、判斷是否觸發 30 天長天期鎖利。

line_notifier.py —— 封裝 LINE Messaging API，負責將機器人成功掛單摘要或系統崩潰訊息推播至您的手機 LINE。

utils/（輔助工具）

logger.py —— 統一的日誌系統，將運行紀錄同時輸出至終端機與本地 .log 檔案，方便維運排錯。

main.py —— 整個專案的進入點，內含 systemd 所守護的 while True 主迴圈。

requirements.txt —— 記錄專案所需的第三方套件依賴清單。
